## 1. 系统概述

### 1.1 系统定位

毛主席生平纪念网站是一个面向公众的公益性红色文化展示平台，采用 B/S（Browser/Server）架构，以 Web 网站形式提供服务。系统分为**前台展示系统**和**后台管理系统**两部分，前台面向所有互联网用户开放，后台仅限管理员访问。

### 2.2 架构设计原则

1. **前后端分离**：前端负责 UI 渲染和交互，后端提供 API 接口，通过 RESTful API 通信。
2. **SSR/SSG 混合渲染**：首页和时间轴页采用 SSG（静态生成）提升性能；搜索和后台采用 SSR/CSR。
3. **缓存优先**：热点数据优先从 Redis 读取，减少数据库压力。
4. **渐进式加载**：时间轴采用懒加载，首屏只加载可视区域内容。
5. **安全第一**：所有输入校验、输出转义、接口鉴权。
6. **可扩展性**：模块化设计，便于后续功能扩展。

### 4.3 各模块职责详述

#### M02 时间轴模块

**职责**：
- 从数据库/缓存读取时间节点数据
- 按时间排序展示节点
- 提供年代筛选功能
- 实现滚动加载和动画效果
- 记录节点浏览量

**核心接口**：
- `GET /api/timeline` - 获取时间轴节点列表（分页）
- `GET /api/timeline/:id` - 获取单个节点详情
- `GET /api/timeline/era/:era` - 按年代筛选节点

**依赖模块**：M11(数据访问)、M12(缓存)、M14(日志)

### 5.2 部署方案

**开发环境**：
- 单机部署：Node.js + MySQL + Redis
- Docker Compose 编排

**测试环境**：
- 单机部署：Docker Compose
- 数据独立，不与生产共享

**生产环境**：
- 应用服务器：2-3 台，Nginx 负载均衡

## 8. 数据架构

### 8.1 数据存储策略

| 数据类型 | 存储位置 | 说明 |
|---------|---------|------|
| 结构化业务数据 | MySQL | 节点、留言、管理员等 |
| 热点缓存数据 | Redis | 时间轴、节点详情 |
| 图片文件 | OSS / 本地 | 历史照片 |
| 日志文件 | 文件系统 | 操作日志、访问日志 |
| 会话信息 | Redis | JWT Token 白名单 |

### 8.2 数据流向

```
数据写入流:
用户操作 → API接口 → 参数校验 → 业务逻辑 → 数据校验
→ 写入MySQL → 清除相关缓存 → 记录日志 → 返回结果

数据读取流:
用户请求 → API接口 → 检查Redis缓存
→ [命中] 直接返回
→ [未命中] 查询MySQL → 写入Redis → 返回结果
```

### 8.3 数据一致性

1. **缓存更新策略**：Cache Aside Pattern（先更新数据库，再删除缓存）
2. **缓存失效**：写入数据时主动清除相关缓存键
3. **最终一致性**：通过消息队列异步更新缓存（可选）

## 附录：架构决策记录（ADR）

### ADR-001: 选择 Next.js 全栈方案而非前后端分离

**背景**：项目需要 SEO 友好且开发效率高。

**决策**：采用 Next.js 全栈方案，前端页面和 API 路由在同一项目中。

**理由**：
1. SSG/SSR 能力满足 SEO 需求
2. 前后端同语言，降低沟通成本
3. 部署简单，单服务即可
4. 社区活跃，文档完善

**影响**：项目结构统一，开发效率提升。

---

### ADR-002: 选择 MySQL 而非 PostgreSQL

**背景**：需要选择关系型数据库。

**决策**：采用 MySQL 8.0。

**理由**：
1. 运维团队更熟悉 MySQL
2. 数据量不大，MySQL 性能足够
3. 云服务商 MySQL 支持更成熟
4. 成本较低

**影响**：需注意 MySQL 的 JSON 字段处理差异。

---

### ADR-003: 选择 Prisma 而非 TypeORM

**背景**：需要 TypeScript 友好的 ORM。

**决策**：采用 Prisma 5+。

**理由**：
1. 类型安全，自动生成类型
2. Schema 优先，迁移管理方便
3. 查询 API 直观，学习成本低
4. 性能优秀

**影响**：开发体验提升，类型检查更严格。
